# Fix Plan — letletme_data (2026-07-17)

Covers **all 64 findings** from [`code-review-2026-07-17.md`](./code-review-2026-07-17.md) (6 Critical · 14 High · 24 Medium · 20 Low), grouped into **24 fix items (FP-01…FP-25)** across three priority tiers.

**Ground rules**

1. **Redis is a shared, external contract.** Multiple known consumer systems read this Redis. Every existing key pattern, hash field, and JSON shape is **frozen**. Fixes must be writer-side or reader-side *within* existing shapes. New data needs → **new additive keys only**; never rename, re-shape, or repurpose. Deletions (season rollover, retention) require consumer sign-off and are manual runbooks, never automatic jobs.
2. **`bun run db:generate` is frozen** until FP-01 merges (it would emit a schema-reset migration that both migration tools would apply).
3. Each FP item = one PR. P0 items land in order; P1 items are parallelizable except where noted.
4. Effort scale: **XS** < 30 min · **S** < 2 h · **M** ≈ half-day · **L** 1–2 days.

**Input still needed from you:** the consumer inventory (which systems read which Redis keys) to complete `docs/redis-contract.md` (FP-06). Until it's provided, the plan treats *all* keys as consumed.
